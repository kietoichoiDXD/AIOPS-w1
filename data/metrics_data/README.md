# Cloud Metrics Time-Series Dataset

> Generated by `generate_metrics.py` — Realistic synthetic cloud infrastructure metrics
> for anomaly detection model training and evaluation.

---

## Purpose

This dataset provides **hourly cloud infrastructure metrics** for 61 resources across
6 AWS accounts over 92 days (2026-03-01 to 2026-05-31). It is designed to:

1. **Train anomaly detection models** that generalize to real-world infrastructure
2. **Avoid label leakage** — metrics are NOT generated from labels
3. **Challenge simplistic detectors** — no single metric perfectly separates classes

### Key Properties

| Property | Value |
|---|---|
| Time range | 2026-03-01 00:00 to 2026-05-31 23:00 (UTC) |
| Granularity | Hourly (2,208 timestamps) |
| Resources | 61 (from 276 in CUR, representative subset) |
| Total rows | 134,688 |
| File size | ~24 MB |
| Label: normal | ~90% |
| Label: anomaly | ~6% |
| Label: benign | ~4% |

## Generation Methodology

### 1. Resource Profiling
Resources are extracted from `cur_line_items.csv`. Average daily costs per resource
determine baseline metric levels (probabilistic, with noise).

### 2. Stochastic Processes

| Process | Used For | Formula |
|---|---|---|
| Ornstein-Uhlenbeck | CPU, memory, DB connections | `dX = theta(mu-X)dt + sigma*dW` (mean-reverting) |
| Geometric Brownian Motion | Network, disk IO | `dX = mu*X*dt + sigma*X*dW` (multiplicative growth) |
| Poisson Burst | Request spikes | Random burst events |

### 3. Seasonality
- **Daily**: Business hours (9-17) have higher utilization; nights are lower
- **Weekly**: Weekdays > Saturday > Sunday
- **Resource-type overrides**: Databases are flatter; GPU training is less periodic

### 4. Cross-Metric Correlations
Correlated noise via Cholesky decomposition:
- CPU <-> Memory (0.65)
- CPU <-> Network (0.45-0.50)
- Network In <-> Network Out (0.60)
- DB Connections <-> CPU (0.55)
- GPU <-> Memory (0.55)

### 5. Growth Trend
Organic growth ~0.08%/day with account-specific modifiers
(data-analytics grows fastest, staging slowest).

## Anomaly Events

Anomalies emerge from **realistic failure mechanisms**, not label injection:

| Event Type | Mechanism | Duration | Primary Metric |
|---|---|---|---|
| `memory_leak` | Linear ramp | 48-168h | memory_mib |
| `ddos_spike` | Exponential spike | 6-48h | network_in_bytes |
| `db_saturation` | Step increase | 12-72h | database_connections |
| `disk_bottleneck` | Saturation plateau | 24-120h | disk_io_ops |
| `gpu_runaway` | Saturation plateau | 48-168h | gpu_utilization |
| `idle_resource` | Suppression | 336h+ | cpu_percent |

Each anomaly has:
- **Onset**: Gradual or sudden start
- **Duration**: Hours to days
- **Recovery**: Gradual return to baseline
- **Secondary effects**: Correlated changes in other metrics

---

## Benign Events

Legitimate operational events that look like anomalies:

| Event Type | Pattern | Periodicity |
|---|---|---|
| `autoscaling` | CPU/memory burst | Daily |
| `scheduled_backup` | Disk IO + network burst at 2 AM | Daily |
| `batch_etl` | Full-stack increase | Weekly (Monday 10 PM) |
| `ml_training` | GPU + CPU + memory increase | Weekly |
| `data_migration` | Network egress burst | One-time (Mar 28-30) |
| `deployment` | CPU + network burst | Biweekly |

## Leakage Prevention

This dataset satisfies all anti-leakage requirements:

1. **No metric encodes labels** — metrics generated first, labels assigned after
2. **No perfect single-feature separation** — all features have distribution overlap
3. **Benign-anomaly overlap** — benign and anomaly distributions partially overlap
4. **Normal spikes** — 0.5% of normal hours contain random spikes
5. **Subtle anomalies** — 30% of anomalies have dampened effects
6. **Extreme benign** — 20% of benign events have amplified effects
7. **Feature importance distributed** — no single metric dominates

### Validation Results

```
[Check 1] Label Distribution
  [OK] normal:  ~90%  (target: 83-92%)
  [OK] benign:  ~4%   (target: 3-12%)
  [OK] anomaly: ~6%   (target: 3-10%)

[Check 2] No Perfect Single-Metric Separation
  [OK] All metrics have normal-anomaly overlap

  [OK] All primary metrics overlap

[Check 5] Feature Importance
  [OK] Max |correlation with anomaly| = 0.09 (well below 0.5 threshold)
```

## Quick Start

```python
import pandas as pd

# Load metrics
metrics = pd.read_csv("metrics.csv", parse_dates=["timestamp"])

# Basic exploration
print(metrics.shape)
print(metrics["label"].value_counts(normalize=True))
print(metrics["resource_type"].value_counts())

# Per-resource time series
resource = metrics[metrics["resource_id"] == "i-0000000003e9"]
resource.set_index("timestamp")[["cpu_percent", "memory_mib"]].plot()

# Join with cost data for enrichment
cur = pd.read_csv("../tf2-finops/cur_line_items.csv")
```

## Connection to FinOps Dataset

This metrics dataset is designed to work alongside the TF2 FinOps dataset:

- **Resource IDs** match `cur_line_items.csv` resource identifiers
- **Account names** match linked accounts (prod-core, prod-payments, staging, dev, data-analytics, ml-research)
- **Time range** aligns exactly (2026-03-01 to 2026-05-31)
- **Known labels** from `anomaly_labels_public.csv` are incorporated:
  - A2 (idle_resource, RDS): Mapped to database resource with suppressed metrics
  - A6 (sudden_spike, CloudWatch): Mapped with exponential spike pattern
  - B2 (benign, data migration): Mapped as one-time network burst

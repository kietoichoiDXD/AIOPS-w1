# ADR-001: Use Kafka for telemetry transport

## Status

Accepted

## Context

The payment service emits metrics, logs, and traces at a rate that can spike during checkout peaks and retry storms. Direct push from services into storage looks simple at first, but it couples application latency to backend availability. If the storage tier slows down, telemetry drops or request paths get slower. That is risky for an AIOps stack because observability must not become part of the outage.

For the target design, we need a transport layer that can buffer bursts, preserve data for replay, and support multiple downstream consumers. The main alternative is direct push from collectors into storage. That option is cheaper to start, but it becomes fragile once the service count grows and one backend gets slow. For example, a burst that adds even 5-20 ms per telemetry write is acceptable for dashboards but not acceptable when the application already sits under checkout pressure.

## Decision

Use Kafka as the telemetry transport between collection and processing.

Services emit telemetry through OpenTelemetry SDKs to OpenTelemetry Collectors. Collectors batch and forward events to Kafka. Downstream consumers such as Flink, log indexing jobs, and archival pipelines read from Kafka independently.

## Consequences

This choice adds operational overhead because we must run and monitor a Kafka cluster. However, it gives us replay, backpressure protection, and decoupling between producers and consumers. In practice, that means a storage outage no longer breaks ingestion; it only increases lag. For a medium deployment, keeping telemetry reliable is worth the extra infrastructure cost because losing one incident window is more expensive than running the broker layer.

The trade-off is that latency is slightly higher than direct push. A direct collector-to-storage path may finish in under 50 ms under normal load, while Kafka typically adds around 5-20 ms. That is acceptable for observability because the system favors durability and fan-out over ultra-low latency.

## Alternatives

- Direct push to storage: simpler, lower latency, but fragile under backend failures.
- NATS: lightweight and fast, but weaker durability defaults for this use case.
- Kinesis: good managed option on AWS, but less portable and less transparent for this assignment.
- Pulsar: strong multi-tenant design, but more complex than needed here.
